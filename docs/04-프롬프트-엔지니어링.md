# 프롬프트 엔지니어링 — 에이전트가 되묻는 문제 해결

## 문제

Discord에서 에이전트에게 메시지를 보내면 분석을 시작하는 대신 되묻기만 했다.

```
[운영자] 서버 상태 어때?

[에이전트] 어떤 서버를 확인해 드릴까요?
          현재 발생 중인 알람이 있나요?
          더 자세한 정보를 알려주시면 도움이 될 것 같습니다.
```

이 응답은 에이전트로서 전혀 유용하지 않다. 도구(`verify_alert`, `query_prometheus`)를 호출하면 즉시 알 수 있는 정보를 사용자에게 묻고 있다.

---

## 원인 분석

### 1. RLHF 정렬 편향

현대 LLM은 RLHF(Reinforcement Learning from Human Feedback)로 학습된다. 이 과정에서 사람 평가자들이 "불확실할 때 질문하는 응답"에 높은 점수를 주는 경향이 있다. 일반 대화에서는 적절한 행동이지만, 에이전트 맥락에서는 문제가 된다.

에이전트는 사용자에게 물어보는 대신 도구로 직접 찾아야 한다.

### 2. 프롬프트 구문의 편향 강화

초기 시스템 프롬프트에 다음과 같은 구문이 있었다.

```
근거가 불충분하면 질문해서 정보를 보완해.
```

이 문구가 RLHF 편향과 결합해 "도구를 쓰기 전에 질문"을 강화했다. LLM 관점에서 도구 호출 전에 질문하는 것이 항상 더 안전한 선택처럼 보이기 때문이다.

### 3. 역할 프레이밍 부재

초기 프롬프트에는 에이전트가 누구인지, 어떤 맥락에서 동작하는지 명확한 역할이 없었다. 역할이 없으면 LLM은 기본 대화 상대 모드로 동작한다.

---

## 연구 근거

단순히 "질문하지 마"라고 추가하는 것은 편협한 해결책이다. 업계 연구를 먼저 검토했다.

**OpenAI GPT-4.1 에이전트 가이드 (2024)**
> "Tool-Over-Ask: 에이전트는 사람에게 묻기 전에 도구로 먼저 찾아야 한다. 도구로 얻을 수 없는 정보일 때만 사용자에게 질문하라."

**G-Eval (Liu et al., Google, 2023)**
모니터링 에이전트 응답의 품질 평가 기준으로 "Tool Appropriateness(도구 사용 적절성)"를 별도 차원으로 제시한다. 도구를 사용하지 않고 질문만 한 응답은 낮은 점수를 받는다.

**PPP Framework (Prompt engineering for autonomous agents)**
역할 프레이밍(Role framing)이 에이전트의 자율적 행동을 유도하는 데 효과적임을 보인다. "1차 대응자"처럼 구체적인 직무 역할을 부여하면 해당 역할의 행동 패턴을 따른다.

---

## 해결: 3가지 변경

### 1. 역할 프레이밍

**Before:**
```
너는 인프라 모니터링 도우미야.
```

**After:**
```
너는 인프라 모니터링 1차 대응자야. 발생한 알람을 자율적으로 분석하고
근본 원인과 해결 방법을 찾아. 조사를 시작하는 데 추가 허락이 필요 없어.
```

"1차 대응자"는 알람을 받으면 즉시 조사를 시작하는 역할이다. 이 역할 프레이밍이 LLM의 행동 방향을 바꾼다.

### 2. Tool-Over-Ask 규칙 명시

**Before:** 없음

**After:**
```
[핵심 원칙: 도구로 찾고, 찾은 뒤에 말해]
- 정보가 부족하면 사람에게 묻지 말고 도구를 호출해서 찾아라.
- 도구로도 얻을 수 없는 정보일 때만 운영자에게 질문해.
- 요청이 모호해도 스스로 판단해서 관련 도구를 먼저 실행해.
  예) "서버 상태 어때?" → verify_alert + query_prometheus 즉시 호출
  예) "네트워크 이상 없어?" → 네트워크 관련 메트릭 즉시 조회
```

### 3. 종료 기준 명시

LLM이 결론을 내리지 않고 데이터를 계속 수집하는 문제도 있었다. 종료 기준을 명시해 해결했다.

**Before:** 없음

**After:**
```
[조사 종료 기준]
- 근본 원인 가설이 확인되거나 배제되면 즉시 결론 내려.
- 같은 방향을 가리키는 데이터 2~3개면 충분해. 더 수집하지 마.
- 정상이면 "정상입니다 + 근거", 이상이면 "무엇이 왜 이상한지" 바로 답해.
```

---

## 결과

변경 후 같은 질문에 대한 응답:

```
[운영자] 서버 상태 어때?

[에이전트] (verify_alert 즉시 호출)
          (query_prometheus CPU/메모리/디스크 조회)

          현재 모든 알람 정상 상태입니다.

          수집한 주요 메트릭:
          - CPU: 12% (prod-server-01)
          - 메모리: 58% (prod-server-01)
          - 디스크: 43%

          이상 징후 없습니다.
```

---

## 교훈

LLM 프롬프트 문제를 디버깅할 때 "이 구문을 추가하면 되겠다"는 직관보다, **왜 이 행동이 나타나는지** 원인을 먼저 파악해야 한다. RLHF 편향을 이해하지 못한 상태에서 패치성 구문을 추가하는 것은 다른 케이스에서 예상치 못한 부작용을 낳는다.

역할 프레이밍, Tool-Over-Ask, 종료 기준 세 가지는 각각 다른 원인을 해결한다.
- 역할 프레이밍: "무엇을 해야 하는가"
- Tool-Over-Ask: "어떻게 행동해야 하는가"
- 종료 기준: "언제 멈춰야 하는가"
