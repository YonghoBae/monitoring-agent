# 에이전트 파이프라인 설계

## 전체 흐름

알람 하나가 들어왔을 때 어떤 과정을 거쳐 Discord에 분석 결과가 도달하는지 순서대로 설명한다.

```
Alertmanager Webhook
      │
      ▼
AlertService.createAlert()
  ├─ AlertEvent 저장 (PostgreSQL)
  ├─ PGVector 인덱싱 (AlertVectorService)
  ├─ AlertVerifier — 알람이 현재도 발생 중인지 확인
  ├─ AlertPlaybook — 알람명으로 허가된 조치 조회
  │
  ├─ [동시 알람 3개 미만] → ReActAgent.run()
  └─ [동시 알람 3개 이상] → OrchestratorAgent.analyze()
                                └─ 그룹 컨텍스트 구성
                                └─ 각 알람에 ReActAgent.runWithContext()
      │
      ▼
   ReActAgent (ReAct 루프)
      ├─ Gemini tool call 루프 (search_rag → verify_alert → query_prometheus/loki)
      └─ 결론 도출
      │
      ▼
   ReflectionAgent (자기 검증)
      ├─ SUFFICIENT → 결론 확정
      └─ INSUFFICIENT → ReActAgent 재실행 (1회 한도)
      │
      ▼
   AgentJudgeEvaluator (품질 평가, 비동기)
      └─ 4개 차원 점수 DB 저장
      │
      ▼
   DiscordNotificationService
      ├─ [NEEDS_APPROVAL] → 분석 결과 + 명령 실행 승인 요청
      ├─ [AUTO] → 명령 자동 실행 후 결과 전송
      └─ [READ_ONLY/NONE] → 분석 결과만 전송
      │
      ▼
   ConversationSessionStore.getOrCreate()
      └─ 알람 채널에 대화 세션 생성 (30분 TTL)
```

## ReAct 에이전트 상세

### ReAct 패턴이란

Reason + Act의 합성어. LLM이 생각(Thought) → 행동(Action) → 관찰(Observation) 사이클을 반복하며 스스로 데이터를 수집하고 결론에 도달한다.

```
[Thought]  CPU 급등 원인 파악을 위해 과거 사례를 먼저 확인한다
[Action]   search_rag("HostHighCpuLoad")
[Observation] "3주 전 배치 작업이 CPU를 90% 점유한 사례 있음"

[Thought]  현재 상황이 같은지 확인한다
[Action]   query_prometheus("node_cpu_seconds_total{instance='prod-server-01'}")
[Observation] "현재 CPU 87%, python3 프로세스가 65% 점유"

[Thought]  배치 작업 여부를 로그에서 확인한다
[Action]   query_loki("{job='python-service'}", "last 10m")
[Observation] "ml-training job 실행 로그 확인"

[결론 도출] 근본 원인: ML 학습 배치 작업 CPU 과점유 ...
```

Spring AI의 `ChatClient`가 이 루프를 내부적으로 처리한다. `@Tool` 어노테이션이 붙은 메서드를 Gemini가 tool call하면, Spring AI가 해당 메서드를 실행하고 결과를 다음 Gemini 호출에 포함시키는 과정을 자동으로 반복한다.

### 도구 우선순위

시스템 프롬프트에 도구 호출 순서를 명시해 불필요한 API 호출을 줄인다.

```
1. search_rag       — 우리 서버 과거 사례 우선
2. verify_alert     — 알람이 현재 발생 중인지 확인
3. query_prometheus — 메트릭 기반 가설 검증
   query_loki       — 로그 기반 가설 검증
4. web_search       — 1~3으로 해결 못할 때만 (최후 수단)
```

### 종료 기준

LLM의 특성상 데이터를 계속 수집하려는 경향이 있다. 이를 막기 위해 종료 조건을 명시한다.

```
[종료 기준]
- 근본 원인 가설이 데이터로 확인되거나 배제됐을 때
- 같은 방향을 가리키는 데이터 포인트 2~3개가 모였을 때
이 기준을 넘어서 계속 수집하지 마.
```

## OrchestratorAgent

동시에 3개 이상의 알람이 발생하면 각각 독립적으로 분석하는 것보다 연관성을 파악하는 것이 중요하다. 예를 들어 CPU, Memory, Disk 알람이 동시에 오면 공통 원인(예: 특정 프로세스 폭주)이 있을 가능성이 높다.

OrchestratorAgent는 다음 컨텍스트를 각 ReActAgent에 주입한다.

```
[동시 발생 알람 그룹]
- HostHighCpuLoad (prod-server-01)
- HostHighMemoryUsage (prod-server-01)
→ 같은 서버에서 CPU와 메모리 동시 급등
→ 두 알람의 공통 원인을 우선 탐색할 것
```

## ReflectionAgent

ReActAgent가 결론을 낸 후, 그 결론이 충분한 근거를 갖추는지 별도의 LLM 호출로 검증한다.

```java
// ReflectionAgent가 평가하는 기준
"INSUFFICIENT: 메트릭 데이터 없이 추측만으로 결론을 내렸다"
"SUFFICIENT: 수집된 데이터로 결론이 충분히 뒷받침된다"
```

`INSUFFICIENT`가 반환되면 해당 피드백을 포함해 ReActAgent를 한 번 더 실행한다. 무한 루프를 막기 위해 재실행은 1회로 제한한다.

## ConversationAgent (양방향 대화)

알람 분석 후 운영자가 Discord에서 추가 질문이나 지시를 할 수 있다.

```
[Discord 채팅]
운영자: "방금 분석한 거 ML 학습 작업 멈추면 어때?"
에이전트: [query_prometheus 즉시 호출]
         "현재 cpu 52%로 내려오고 있습니다. ml-training은 30분 후 완료 예정 ..."
```

**대화 히스토리 유지**: Spring AI의 `MessageChatMemoryAdvisor`가 채널별 대화 내역을 메모리에 보관한다. 같은 채널의 이전 대화가 다음 LLM 호출의 컨텍스트에 자동으로 포함된다.

```java
MessageChatMemoryAdvisor memoryAdvisor = MessageChatMemoryAdvisor
    .builder(session.chatMemory())          // InMemoryChatMemoryRepository
    .conversationId(session.channelId())    // Discord 채널 ID로 구분
    .build();
```

**세션 관리**: 채널별 `ConversationSession`이 30분 TTL로 관리된다. 알람 발생 시 해당 채널에 자동으로 세션이 생성되고, 마지막 메시지로부터 30분이 지나면 세션이 만료된다.
