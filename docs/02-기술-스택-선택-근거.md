# 기술 스택 선택 근거

각 기술을 선택할 때 어떤 대안을 고려했고, 왜 이 선택을 했는지 기록한다.

---

## Spring AI vs LangChain4j vs 직접 API 호출

### 고려한 대안
- **LangChain4j**: Java 생태계의 LangChain 포팅 라이브러리
- **직접 API 호출**: Gemini REST API를 HttpClient로 직접 호출

### 선택: Spring AI

**Function Calling 추상화**: 에이전트가 도구를 호출하는 ReAct 패턴을 구현하려면 LLM의 Function Calling 결과를 받아 실제 메서드를 호출하고 결과를 다시 LLM에 넘기는 루프가 필요하다. Spring AI의 `ChatClient`는 이 루프를 내부적으로 처리한다.

```java
// Spring AI: 도구 등록만 하면 Gemini tool call 루프를 자동 처리
String conclusion = chatClient.prompt()
    .system(systemPrompt)
    .user(userMessage)
    .tools(agentTools, webSearchTool)  // @Tool 어노테이션으로 등록
    .call()
    .content();
```

직접 API 호출로 구현하면 tool call → 결과 파싱 → 다음 요청 구성 루프를 수동으로 작성해야 한다. 코드량과 복잡도가 크게 늘어난다.

**Spring 생태계 통합**: `@Service`, `@Value`, `@Qualifier` 등 기존 Spring 패턴 그대로 사용 가능하다. `@ConditionalOnProperty`로 API 키 없을 때 빈 생성을 건너뛰는 것도 자연스럽게 처리된다.

**LangChain4j를 선택하지 않은 이유**: Spring AI가 Spring Boot Auto-configuration을 공식 지원하고, Spring Boot 3.x와의 호환성이 더 안정적이다. LangChain4j는 별도 Spring 통합 모듈을 써야 한다.

## ReAct 패턴 vs 단순 LLM 호출

### 고려한 대안
- **단순 LLM 호출**: 알람 정보를 그대로 LLM에 넘기고 분석 결과를 받는다
- **Chain of Thought(CoT)**: 단계별 추론을 프롬프트로 유도

### 선택: ReAct (Reason + Act)

단순 LLM 호출의 근본적 한계는 LLM이 학습 시점 이후의 **실시간 데이터에 접근할 수 없다**는 것이다. 알람 분석에는 현재 메트릭, 현재 로그, 우리 서버의 과거 해결 사례가 필요하다. 이 정보는 LLM이 가질 수 없다.

ReAct 패턴은 LLM이 스스로 어떤 데이터가 필요한지 판단하고 도구를 호출해 가져온다.

```
Thought: "CPU 급등 원인을 찾으려면 현재 메트릭을 봐야 한다"
Action: query_prometheus("node_cpu_seconds_total")
Observation: "CPU 85%, 특정 프로세스가 60% 점유 중"
Thought: "과거에 유사한 사례가 있었는지 확인한다"
Action: search_rag("CPU spike process")
Observation: "3주 전 배치 작업이 원인이었던 사례 있음"
→ 결론 도출
```

CoT만으로는 "추론 과정"만 개선되고 실시간 데이터 부재 문제는 해결되지 않는다.

**추가 검증 계층**: ReAct 루프가 끝나면 ReflectionAgent가 결론을 검토한다. 근거가 부족하면 한 번 더 ReAct 루프를 실행한다. 자기 검증 구조로 오탐 가능성을 줄인다.

## PGVector vs Chroma vs Pinecone

### 고려한 대안
- **Chroma**: 경량 벡터 DB, Python 중심 생태계
- **Pinecone**: 관리형 클라우드 벡터 DB

### 선택: PGVector (PostgreSQL 확장)

**추가 인프라 없음**: 이미 PostgreSQL을 사용하고 있어 PGVector 확장만 추가하면 된다. Chroma나 Pinecone을 추가하면 관리해야 할 서비스가 하나 더 늘어난다.

**트랜잭션 일관성**: 알람 해결 피드백을 받았을 때 `ResolutionRecord`를 PostgreSQL에 저장하고 PGVector에 임베딩을 저장하는 두 작업이 같은 DB 트랜잭션에 있다. 벡터 DB를 별도로 두면 두 저장소 간 정합성 관리가 복잡해진다.

**Spring AI 지원**: Spring AI 1.1.0이 `spring-ai-starter-vector-store-pgvector`를 공식 지원한다. 설정 몇 줄로 `VectorStore` 빈을 주입받을 수 있다.

## JDA vs Discord4J vs Javacord

### 고려한 대안
- **Discord4J**: Reactor 기반 리액티브 Discord 라이브러리
- **Javacord**: 경량 Discord 라이브러리

### 선택: JDA (Java Discord API)

**문서화 수준**: JDA는 GitHub Stars, JavaDoc 품질, Stack Overflow 답변 수 모두 세 라이브러리 중 가장 높다. 트러블슈팅 시 레퍼런스를 찾기 쉽다.

**이벤트 기반 구조**: `ListenerAdapter`를 상속해 필요한 이벤트 핸들러만 오버라이드하는 구조가 Spring의 컴포넌트 모델과 잘 맞는다.

```java
@Component
public class DiscordBotListener extends ListenerAdapter {
    @Override
    public void onMessageReceived(MessageReceivedEvent event) { ... }
}
```

**Discord4J를 선택하지 않은 이유**: Reactor 기반이라 Spring MVC(동기) 코드베이스와 섞이면 `subscribe()`, `block()` 혼재로 복잡해진다. 이 프로젝트는 단독 운영자 대상 소규모 시스템이라 리액티브 스트림의 이점이 없다.

## Google Gemini vs GPT-4 vs Claude

### 선택: Google Gemini (gemini-2.5-flash)

**Function Calling 품질**: 에이전트 패턴의 핵심은 LLM이 적절한 도구를 적절한 순서로 호출하는 것이다. Gemini 2.5-flash는 Function Calling 정확도가 높고, 여러 도구를 순차적으로 호출하는 루프를 안정적으로 처리한다.

**비용**: gemini-2.5-flash는 입력 토큰당 가격이 GPT-4o 대비 낮다. 알람마다 Prometheus/Loki 조회 결과를 컨텍스트에 포함하면 토큰 사용량이 빠르게 늘어나는데, 비용 부담이 적다.

**Spring AI 통합**: `spring-ai-starter-model-google-genai` 의존성 하나로 `@Qualifier("googleGenAiChatModel")` 주입이 가능하다.

## Java 21

Spring Boot 3.5.7이 Java 21을 기본 지원 버전으로 사용한다. 초기에는 Java 17로 시작했으나, 이후 **Java 21로 마이그레이션**했다.

**이유**: Spring AI 1.1.0의 일부 기능이 Java 21에서만 동작하며, Virtual Thread 지원으로 Blocking I/O(Prometheus HTTP 요청, Loki HTTP 요청) 처리 효율이 개선된다. 또한 Record 패턴 매칭, Sequenced Collections 등 Java 21의 언어 기능을 활용할 수 있다.

변경 범위는 Dockerfile 2줄, build.gradle 1줄로 최소화됐다.
