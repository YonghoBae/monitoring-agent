# 에이전트 성능 평가

## 배경

에이전트의 프롬프트나 도구 로직을 수정했을 때 "개선됐는지"를 어떻게 측정할 것인가? 수동으로 결과를 보고 판단하는 것은 주관적이고 반복하기 어렵다.

두 가지 접근을 검토했다.

1. **LLM-as-a-Judge**: 별도 LLM이 에이전트 응답 품질을 자동 채점
2. **PPL (Perplexity)**: 모델의 응답 불확실성 측정

---

## PPL 적용 불가 이유

PPL은 모델이 각 토큰을 얼마나 확신하는지를 나타내는 지표로, 낮을수록 모델이 응답에 확신이 있다는 의미다.

PPL을 측정하려면 각 토큰의 log probability가 필요하다. 그런데 **Gemini API는 logprob를 제공하지 않는다.** Spring AI의 `ChatResponse`에서 접근 가능한 것은 토큰 사용량(입출력 토큰 수)뿐이다.

```java
// Spring AI 1.1.0에서 접근 가능한 메타데이터
var usage = response.getMetadata().getUsage();
usage.getPromptTokenCount();       // 입력 토큰 수 ✅
usage.getCompletionTokenCount();   // 출력 토큰 수 ✅
// logprob, confidence score    → 미제공 ❌
```

GPT-4, Claude API도 logprob를 완전히 노출하지 않는다. 상용 LLM API 환경에서 PPL 직접 측정은 현실적으로 불가능하다.

대신 토큰 수 비율(입력 대비 출력 토큰 비)을 "장황성 지표"로 간접 활용할 수 있다. 응답이 과도하게 길면 모델이 불확실해 장황하게 쓰는 경향이 있다.

## LLM-as-a-Judge 구현

### 방법론

G-Eval(Liu et al., Google, 2023) 방식을 채택했다. 별도의 LLM이 미리 정의된 루브릭(rubric)으로 응답을 채점한다.

```
평가 대상 응답 → Judge LLM (Gemini) → 4개 차원 점수 (1-10) → DB 저장
```

### 4개 평가 차원

| 차원 | 설명 | 10점 기준 | 1점 기준 |
|------|------|---------|---------|
| **Factuality** | 수집한 데이터로 결론이 뒷받침되는가 | 모든 주장에 데이터 근거 있음 | 데이터 없이 추측만 나열 |
| **Tool Use** | 도구를 적절히 사용했는가 (과다/과소 모두 감점) | 최소한의 도구로 필요한 정보 수집 | 중요 도구 누락 또는 과도한 반복 |
| **Actionability** | 권고 조치가 실행 가능한가 | 구체적이고 즉시 실행 가능한 항목 2~3개 | "더 조사 필요" 같은 모호한 조치 |
| **Hallucination Risk** | 확인되지 않은 주장이 없는가 | 모든 주장이 데이터로 검증됨 | 확인되지 않은 단정적 주장 포함 |

### 구현

```java
// AgentJudgeEvaluator.java
// ReActAgent.run() 완료 직후 자동 호출
@Nullable
public AgentEvaluation evaluate(AlertEvent alert, AgentResult agentResult) {
    String judgeResponse = judgeClient.prompt()
        .system(buildJudgeSystemPrompt())   // 4개 차원 루브릭
        .user(buildEvaluationPrompt(alert, agentResult))
        .call()
        .content();

    int[] scores = parseScores(judgeResponse);  // 정규식으로 "X/10" 파싱
    return evaluationRepository.save(new AgentEvaluation(...));
}
```

Judge 프롬프트는 응답 형식을 엄격하게 지정해 파싱 실패를 최소화한다.

```
Factuality: X/10
[이유 한 문장]

Tool Use: X/10
[이유 한 문장]
...
```

파싱 실패 시 해당 차원은 기본값 5점으로 처리한다. Judge 평가 실패가 에이전트 분석 결과에 영향을 주지 않도록 예외를 별도로 처리한다.

### 데이터 저장

```java
// AgentEvaluation 엔티티 (PostgreSQL)
@Entity
public class AgentEvaluation {
    @Version
    private Long version;               // 낙관적 락 (Reflection 재분석 시 중복 저장 방지)
    private int factualityScore;        // 1-10
    private int toolUseScore;           // 1-10
    private int actionabilityScore;     // 1-10
    private int hallucinationRiskScore; // 1-10
    private double overallScore;        // 4개 평균
    private String judgeFeedback;       // judge 상세 피드백 (TEXT)
    private LocalDateTime evaluatedAt;
}
```

### 비용 제어

Judge 평가는 매 분석마다 추가 LLM 호출이 발생해 토큰 비용이 약 2배 증가한다. 기본값은 비활성화다.

```yaml
# application.yml 기본값
agent:
  evaluation:
    enabled: ${AGENT_EVALUATION_ENABLED:false}
```

평가가 필요할 때(예: 프롬프트 변경 전후 비교)만 환경변수로 활성화한다.

```bash
AGENT_EVALUATION_ENABLED=true docker compose up -d
```

## 고정 시나리오 테스트

프롬프트 변경 전후를 비교하려면 **동일한 입력에 대한 비교**가 필요하다. 실제 알람은 통제가 되지 않으므로 고정 시나리오를 미리 정의했다.

### 시나리오 목록 (`eval-scenarios.json`)

| ID | 알람 | 검증 항목 |
|----|------|---------|
| `cpu-spike-high` | HostHighCpuLoad (CPU 80%+) | Prometheus 사용 여부, 재시작 비권고 |
| `container-oom-killed` | ContainerOOMKilled (OOM 종료) | 재시작 승인 절차 권고, 메모리 원인 분석 권고 |
| `gpu-high-temperature` | GPUHighTemperature (85°C+) | Prometheus 사용 여부 |
| `container-down` | ContainerDown (컨테이너 중단) | Loki 로그 확인, 재시작 권고 |
| `apollo-frame-latency` | ApolloGameHighFrameLatency (레이턴시 급증) | Prometheus 사용, 세션 수 확인 |

각 시나리오에는 `expectedCriteria`가 정의되어 있다.

```json
{
  "id": "cpu-spike-high",
  "expectedCriteria": {
    "shouldUsePrometheus": true,
    "shouldNotRecommendRestart": true,
    "maxToolCalls": 6,
    "minJudgeScore": 6
  }
}
```

각 시나리오는 알람 정보, Mock 응답(Prometheus/Loki/RAG), 도구 호출 순서, 기대 기준을 모두 JSON에 담는다.

```json
{
  "id": "S1",
  "alert": { "level": "WARNING", "alertName": "HostHighCpuLoad", ... },
  "mocks": {
    "ragResults": [],
    "verifyAlert": { "alertName": "HostHighCpuLoad", "value": "85.3", "timestamp": "..." },
    "prometheusResponses": [
      { "keyword": "cpu", "responses": ["03:00~03:40 정상 범위...", "java(pid 1823): 38.2%..."] }
    ],
    "lokiResponse": null
  },
  "simulation": {
    "toolCalls": [
      { "tool": "search_rag", "query": "HostHighCpuLoad CPU 과부하 원인" },
      { "tool": "verify_alert", "alertName": "HostHighCpuLoad", ... },
      { "tool": "query_prometheus", "promql": "...", "timeRange": "30" }
    ],
    "conclusion": "근본 원인: ...",
    "recommendation": { "category": "READ_ONLY" },
    "judgeResponse": "Factuality: 8/10\n..."
  },
  "expectedCriteria": {
    "shouldUsePrometheus": true,
    "shouldNotRecommendRestart": true,
    "maxToolCalls": 6,
    "minJudgeScore": 6
  }
}
```

`AgentScenarioSimulationTest`는 JSON을 읽어 모든 시나리오를 `@ParameterizedTest`로 실행하는 제네릭 러너다. **새 시나리오 추가 = JSON 블록 1개 추가**, Java 코드 수정 불필요.

### 활용 방법

1. **Before 측정**: 프롬프트 변경 전 `./gradlew test --tests "*AgentScenarioSimulationTest"` 실행 → Judge 점수 기록
2. **프롬프트 변경**
3. **After 측정**: 동일 시나리오 재실행 → Judge 점수 비교

이 방식으로 "에이전트가 개선됐다"는 주관적 판단을 정량적으로 뒷받침할 수 있다.

## 한계 및 고려사항

**Judge bias**: Judge LLM도 Gemini를 사용한다. 동일 모델이 자신의 응답을 평가하는 구조여서 관대하게 채점할 수 있다(self-serving bias). 이상적으로는 다른 모델(예: GPT-4)을 Judge로 사용하는 것이 더 객관적이다.

**비용**: 매 분석마다 Judge LLM 호출이 추가된다. 토큰 비용이 2배에 가까워진다. 운영 환경에서는 샘플링(예: 10건 중 1건만 평가)으로 비용을 줄일 수 있다.

**재현성**: LLM 응답은 비결정적이다. 동일 시나리오를 여러 번 실행하면 점수가 달라진다. 신뢰할 수 있는 측정을 위해서는 시나리오당 3~5회 실행 후 평균을 사용해야 한다.
