# 인프라 모니터링 자동화 에이전트

## 배경 및 문제 인식

서버를 운영하다 보면 Prometheus/Grafana가 알림을 보내는 순간부터 실제 조치까지 여러 단계를 수동으로 거쳐야 한다.

1. 알림 수신 (Discord / 이메일)
2. 해당 서버 접속 → 메트릭 확인 (Grafana)
3. 로그 확인 (Loki / journalctl)
4. 과거 유사 장애 기억 더듬기
5. 조치 수행

이 과정에서 발생하는 문제는 두 가지다.

- **반응 속도**: 자고 있거나 자리를 비웠을 때 알림을 놓친다
- **컨텍스트 스위칭**: 알림 → 분석 → 조치 각 단계마다 도구를 바꿔가며 작업해야 한다

이를 해결하기 위해 **Alertmanager 웹훅을 수신해 AI가 자율적으로 분석하고 Discord를 통해 결과와 조치를 제안하는 에이전트**를 구축했다.

## 전체 아키텍처

```
[인프라 서버]
  └─ Prometheus (메트릭 수집)
  └─ Loki (로그 수집)
  └─ Alertmanager (임계값 초과 시 알림 발행)
        │
        │ Webhook POST /api/alerts
        ▼
[monitoring-agent (Spring Boot)]
  ├─ AlertService (알람 수신 및 파이프라인 진입)
  │    └─ AlertVerifier (현재 알람이 실제 발생 중인지 검증)
  │    └─ AlertPlaybook (알람별 허가된 조치 목록 조회)
  │
  ├─ 에이전트 파이프라인
  │    └─ OrchestratorAgent (동시 알람 3개 이상 시 그룹 분석)
  │    └─ ReActAgent (Gemini + Function Calling 기반 자율 분석)
  │    └─ ReflectionAgent (분석 결론 자기 검증)
  │    └─ AgentJudgeEvaluator (LLM-as-a-Judge 품질 평가)
  │
  ├─ 도구 (에이전트가 호출)
  │    └─ PrometheusQueryService (메트릭 조회)
  │    └─ LokiQueryService (로그 조회)
  │    └─ AlertVectorService (PGVector RAG 검색)
  │    └─ AlertVerifier (알람 검증)
  │    └─ WebSearchTool (웹 검색, 최후 수단)
  │
  └─ Discord Bot (JDA)
       └─ 분석 결과 전송
       └─ 명령 실행 제안 (docker restart) → 운영자 yes/no 확인
       └─ 양방향 대화 (ConversationAgent)
       └─ 해결 피드백 수신 → PGVector 학습

[PostgreSQL + PGVector]
  └─ AlertEvent (알람 이력)
  └─ ResolutionRecord (해결 기록 → RAG 학습 데이터)
  └─ AgentEvaluation (에이전트 응답 품질 평가 이력)

[Ollama (로컬)]
  └─ all-minilm (임베딩 모델, PGVector 인덱싱용)
```

## 핵심 기능

### 1. 자율 알람 분석
Alertmanager에서 웹훅이 오면 에이전트가 자동으로 Prometheus 메트릭, Loki 로그, 과거 유사 사례(RAG)를 조회하고 근본 원인 및 조치를 분석한다.

### 2. ChatOps — Discord 양방향 대화
분석 결과를 Discord로 전송한 뒤, 운영자가 추가 질문이나 지시를 Discord 채팅으로 할 수 있다. 에이전트는 대화 히스토리를 유지하며 맥락 있는 응답을 한다.

### 3. 안전한 명령 실행 (승인 워크플로우)
에이전트가 `docker restart <container>` 실행이 필요하다고 판단하면 직접 실행하지 않고 Discord로 확인 요청을 보낸다. 운영자가 `yes`를 입력해야 실행된다. 허가된 명령 패턴(`docker restart <name>`)만 실행 가능하다.

### 4. RAG 기반 과거 사례 학습
운영자가 `resolved <alertId> <해결방법>` 또는 `false-positive <alertId>`를 Discord에 입력하면 해당 알람의 해결 과정이 PGVector에 임베딩되어 저장된다. 이후 유사한 알람 발생 시 에이전트가 과거 사례를 먼저 참조한다.

### 5. LLM-as-a-Judge 품질 평가
매 분석 후 별도의 Judge LLM이 응답 품질을 4개 차원으로 자동 채점하고 DB에 저장한다. 프롬프트 개선의 정량적 근거로 활용한다.
