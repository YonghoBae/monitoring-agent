# 에이전트가 도구 대신 질문을 했다

## 도구를 줬는데 왜 쓰지 않는가

ReAct 루프를 구현하고 처음 돌려봤을 때, 에이전트는 도구를 거의 쓰지 않았다. `verify_alert`, `query_prometheus`, `query_loki`를 모두 등록해줬는데도, 나오는 답변은 이런 식이었다.

> "현재 ContainerDown 알림이 발생했습니다. 정확한 분석을 위해 해당 컨테이너의 현재 상태와 로그를 확인해주세요."

도구를 호출하지 않고, 운영자에게 확인을 요청했다. 알림 메시지를 그대로 요약해서 돌려준 것과 다를 게 없었다. 운영자가 직접 확인해야 한다면 에이전트가 왜 필요한가.

## 왜 이런 일이 생기는가

처음엔 Spring AI나 Gemini function calling의 문제라고 생각했다. 그런데 도구 설명을 바꿔봐도, 도구 개수를 줄여봐도 현상이 반복됐다.

조금 더 관찰해보니 패턴이 보였다. 에이전트는 "데이터가 부족해서 판단을 못하겠다"고 말하면서 도구를 쓰지 않았다. 데이터가 부족하면 도구를 써서 가져와야 할 텐데, 대신 사람에게 물었다.

원인은 모델 자체의 훈련 방향에 있었다. RLHF(인간 피드백 강화학습)로 정렬된 모델은 "확실하지 않으면 사람에게 물어라"는 방향으로 학습된다. 이 성향이 에이전트 컨텍스트에서도 그대로 나타난 것이다. 에이전트가 자율적으로 행동하길 원한다면, 그에 맞는 명시적인 지시가 필요했다.

거기에 더해, 초기 시스템 프롬프트에 두 가지가 빠져 있었다.

첫째, 역할이 없었다. 에이전트가 어떤 입장에서 행동해야 하는지 정해주지 않았다. 역할이 없으면 모델은 기본 성향대로 행동한다.

둘째, 언제 멈춰야 하는지 없었다. 종료 기준이 없으면 "더 확인해야 할 것 같다"는 판단이 반복되면서 루프가 돌거나, 반대로 데이터 수집을 시작하기도 전에 멈춰버렸다.

## 세 가지 변경

### 1. 역할 프레이밍

에이전트를 "인프라 모니터링 1차 대응자"로 정의했다.

```
너는 인프라 모니터링 1차 대응자야. 발생한 알람을 자율적으로 분석하고 근본 원인과 해결 방법을 찾아.
알람 정보(레벨, 이름, 레이블, 요약)는 이미 제공되어 있어. 조사를 시작하는 데 추가 허락이 필요 없어.
```

"1차 대응자"는 현장에서 즉각 행동하는 역할이다. 보고하거나 확인을 기다리는 역할이 아니다. "추가 허락이 필요 없어"는 에이전트가 질문 없이 도구를 호출하도록 명시적으로 허용한 것이다. 이 한 줄이 의외로 효과가 있었다.

### 2. 도구 우선 규칙

역할만으로는 부족했다. "정보가 부족하면 어떻게 하라"는 규칙을 추가했다.

```
[도구 호출 순서]
1. search_rag — 우리 서버 과거 사례 먼저 확인
2. verify_alert — 알람이 현재도 발생 중인지 확인
3. query_prometheus / query_loki — 가설 검증에 필요한 메트릭·로그만 선택 조회
4. web_search — 1~3으로 해결 방법을 못 찾았을 때만 사용하는 최후의 수단
```

ConversationAgent에는 더 구체적인 예시를 넣었다.

```
[핵심 원칙: 도구로 찾고, 찾은 뒤에 말해]
- 정보가 부족하면 사람에게 묻지 말고 도구를 호출해서 찾아라.
- 도구로도 얻을 수 없는 정보일 때만 운영자에게 질문해.
- 요청이 모호해도 스스로 판단해서 관련 도구를 먼저 실행해.
  예) "서버 상태 어때?" → verify_alert + query_prometheus 즉시 호출
  예) "네트워크 이상 없어?" → 네트워크 관련 메트릭 즉시 조회
```

추상적인 원칙보다 구체적인 예시가 더 효과적이었다. "어때?"라는 모호한 질문에도 어떤 도구를 써야 하는지 보여줬더니, 에이전트가 유사한 상황에서 도구를 먼저 호출하기 시작했다.

### 3. 종료 기준

역할과 도구 우선 규칙을 추가하자 이번엔 반대 문제가 생겼다. 에이전트가 도구를 너무 많이 호출했다. "확인이 됐다"고 판단하는 기준이 없으니, "한 번 더 확인해두는 게 좋겠다"는 방향으로 계속 돌았다.

명시적인 종료 기준을 추가했다.

```
[종료 기준: 이 조건을 만족하면 즉시 결론 내려]
- 근본 원인 가설이 데이터로 확인되거나 배제됐을 때
- 같은 방향을 가리키는 데이터 포인트 2~3개가 모였을 때
이 기준을 넘어서 계속 수집하지 마. 데이터가 많다고 결론이 좋아지지 않아.
```

"데이터가 많다고 결론이 좋아지지 않아"는 직접 관찰에서 나온 문장이다. 에이전트가 같은 종류의 메트릭을 시간대를 바꿔가며 여러 번 조회하는 걸 보고 추가했다.

## 변경 전후

**변경 전:**
```
에이전트 출력:
"ContainerDown 알림이 발생했습니다. 해당 컨테이너의 현재 상태를 확인해주세요.
Prometheus에서 컨테이너 상태를 조회하거나 직접 docker ps 명령으로 확인하실 수 있습니다."

도구 호출 횟수: 0
```

**변경 후:**
```
[도구 호출 #1] verify_alert — nginx, CONFIRMED (firing 중)
[도구 호출 #2] search_rag — "nginx 컨테이너 다운" — 과거 사례: OOMKilled (2회)
[도구 호출 #3] query_loki — nginx, 최근 30분 — OOMKilled: 메모리 제한 초과 확인

에이전트 출력:
"1) 근본 원인: nginx 컨테이너 메모리 제한 초과 (OOMKilled)
 2) 상황 분석: 현재도 firing 중. 최근 30분 로그에서 OOMKilled 확인. 과거 2회 동일 패턴.
 3) 즉시 조치: docker restart nginx 후 메모리 사용량 모니터링 / memory limit 상향 검토"

도구 호출 횟수: 3
```

## 한계

세 가지 변경으로 주요 문제는 해결됐지만, 종료 기준이 항상 정확히 지켜지진 않는다. "2~3개"라는 숫자가 모델에게 어느 정도 가이드는 되지만, 모호한 상황에서는 에이전트가 스스로 "아직 부족하다"고 판단하고 추가 조회를 시도한다.

이 문제를 완전히 해결하지는 못했다. 대신 Reflection 단계에서 결론이 충분한지 사후 검증하고, 부족하면 피드백을 줘서 한 번 더 돌리는 방식으로 보완했다. 프롬프트로 완벽한 종료 기준을 강제하는 건 현실적으로 어렵고, 자기검증 레이어를 추가하는 게 더 현실적인 접근이라고 판단했다.
