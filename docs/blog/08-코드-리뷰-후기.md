# 코드 리뷰에서 발견한 것들

프로젝트가 어느 정도 돌아가기 시작하면 기능 추가보다 먼저 해야 할 일이 생긴다. 지금까지 빠르게 만들면서 쌓인 것들을 들여다보는 일이다.

기능 단위로 리뷰하면서 발견한 문제들을 정리했다. 큰 설계 문제부터 한 줄 수정으로 끝나는 것까지 섞여 있다.

---

## Log Injection — LLM 출력을 신뢰하면 안 된다

`query_loki`는 Gemini가 결정한 컨테이너 이름으로 Loki에서 로그를 조회하는 Tool이다.

```java
// 수정 전
String logql = "{job=\"docker\"} |= `" + containerName + "`";
```

`containerName`은 Gemini가 분석 중에 결정해서 넘기는 값이다. 검증이 없으면 백틱(`` ` ``)이 포함된 값이 들어왔을 때 LogQL 쿼리가 깨진다.

```
containerName = "nginx` or true`"
→ {job="docker"} |= `nginx` or true`   ← 모든 로그가 조회됨
```

Gemini는 외부 알람 메시지의 영향을 받는다. 알람 메시지에 악의적인 지시문이 포함되면 Gemini가 이상한 `containerName`을 생성할 수 있다 — Prompt Injection이다.

수정 방향은 두 단계로 잡았다.

**1차: 실행 중인 컨테이너 목록 대조**

`docker ps`로 현재 실행 중인 컨테이너 목록을 가져와서 대조한다. 목록에 없는 이름이면 조회 자체를 거부한다.

```java
List<String> running = commandExecutorService.listContainers();
if (!running.isEmpty() && !running.contains(containerName)) {
    return "존재하지 않는 컨테이너: " + containerName;
}
```

`running.isEmpty()`인 경우(docker 데몬 장애 등)는 거부하지 않고 2차로 넘긴다. 모니터링 에이전트 특성상 docker가 죽어있는 상황이 오히려 조사가 필요한 상황이라, 로그 조회를 완전히 막으면 역효과다.

**2차: 정규식 검증 (defense-in-depth)**

```java
Pattern.compile("^[a-zA-Z0-9][a-zA-Z0-9_.\\-]*$")
```

1차 통과 후에도 메타문자가 있으면 차단한다. `CommandExecutorService`에 이미 있던 패턴을 재사용했다.

그리고 Gemini가 로그 조회 전에 컨테이너 목록을 확인할 수 있도록 `get_containers()` Tool도 추가했다. 목록을 보고 고르면 존재하지 않는 이름을 넣을 가능성이 줄어든다.

## OrchestratorAgent — 두 가지 개선

### Structured Output

Gemini에게 알람 그룹핑을 요청할 때 응답 형식을 문자열로 강제하고 있었다.

```
응답은 반드시 다음 형식만 사용해:
GROUP_0: 0,2,3
GROUP_1: 1,4
```

Gemini가 이 형식을 정확히 지키면 괜찮다. 하지만 앞뒤에 설명을 붙이거나 형식이 조금이라도 다르면 파싱이 실패하고 전체를 단일 그룹으로 폴백한다. "모든 알람을 하나의 그룹으로 묶어서 처리"하는 것은 OrchestratorAgent가 없는 것과 같다.

Spring AI의 Structured Output을 쓰면 이 문제가 없어진다.

```java
private record GroupingResult(List<List<Integer>> groups) {}

GroupingResult result = chatClient.prompt()
        .user(prompt)
        .call()
        .entity(GroupingResult.class);
```

Spring AI가 내부적으로 JSON 스키마를 주입하고 응답을 파싱해서 Java 객체로 변환한다. 수동 파싱 코드가 필요 없고, 파싱 실패 가능성도 없다.

### Workers 병렬 실행

알람을 그룹으로 나눈 후 각 그룹을 ReActAgent로 분석하는 부분이 순차 실행이었다.

```java
// 수정 전
for (List<AlertEvent> group : groups) {
    AgentResult result = reActAgent.runWithContext(...);
    ...
}
```

그룹이 3개면 분석 1 → 분석 2 → 분석 3을 기다린다. 각 그룹은 서로 독립적이므로 병렬로 실행해야 한다. Orchestrator-Workers 패턴의 핵심 이점이 병렬 처리인데, 구현에서 빠져있었다.

```java
// 수정 후
List<CompletableFuture<AgentResult>> futures = groups.stream()
        .map(group -> CompletableFuture.supplyAsync(() ->
                reActAgent.runWithContext(...)))
        .toList();

List<AgentResult> results = futures.stream()
        .map(CompletableFuture::join)
        .toList();
```

그룹 수가 늘어도 가장 오래 걸리는 그룹 하나만큼의 시간으로 줄어든다.

## Judge 평가 — 비동기로 분리했어야 했다

`judgeEvaluator.evaluate()`는 Gemini를 한 번 더 호출해서 분석 결과를 채점하는 작업이다.

```java
// 수정 전
public AgentResult run(AlertEvent alert, ActionRecommendation recommendation) {
    AgentResult result = runInternal(...);
    judgeEvaluator.evaluate(alert, result);  // 채점 끝날 때까지 블로킹
    return result;
}
```

채점은 나중에 볼 평가 데이터다. 알람 분석 결과를 Discord로 보내는 데 채점 완료가 필요하지 않다. 그런데 동기로 실행되고 있었다.

```java
// 수정 후
@Async
public void evaluate(AlertEvent alert, AgentResult agentResult) { ... }
```

`@Async`를 붙이고 반환값을 `void`로 바꿨다. 호출하는 쪽에서 반환값을 쓰지 않으므로 변경이 없다. 분석 결과는 채점 완료를 기다리지 않고 바로 반환된다.

그리고 같은 리뷰 중에 버그도 발견했다. `run()`은 Judge 평가를 호출하지만 `runWithContext()`는 호출하지 않고 있었다. OrchestratorAgent가 발동되면 Judge 평가가 전혀 기록되지 않는 상태였다. 비동기 전환하면서 같이 수정했다.

## 작은 것들

**`AlertEvent.createdAt` — `@PrePersist`로 보장**

생성자가 두 개인데 둘 다 `this.createdAt = LocalDateTime.now()`를 직접 세팅하고 있었다. 생성자가 추가될 때 누락될 위험이 있다. `@PrePersist`로 JPA가 저장 시점에 보장하도록 바꿨다. `createdAt`은 "저장된 시각"이므로 DB 저장 시점에 세팅되는 게 의미상으로도 맞다.

**`ConversationSessionStore` — 설정 키 재사용**

대화 세션 TTL을 승인 대기 TTL 설정값(`discord.bot.approval-ttl-minutes`)에서 읽고 있었다. 승인 대기와 대화 세션은 다른 개념인데 같은 설정을 공유하면, 하나를 바꿀 때 의도치 않게 다른 쪽도 바뀐다. `discord.bot.session-ttl-minutes`로 분리했다.

**`AgentResult.recommendation` — 읽는 곳이 없는 필드**

`AgentResult`에 `recommendation` 필드가 있었다. 에이전트가 분석 결과와 함께 권장 조치를 반환하는 설계였는데, `AlertAnalysisChain` 리팩토링 이후 호출하는 쪽이 항상 Playbook에서 직접 꺼내 쓰기 때문에 `agentResult.recommendation()`을 읽는 곳이 없었다. 필드와 관련 생성자 파라미터를 제거했다.

---

기능은 그대로인데 코드가 더 단순해졌다. 리뷰 전에는 로직이 얽혀 있어서 흐름을 따라가기가 어려웠는데, 각 부분이 하나의 책임만 갖도록 정리하고 나니 변경 범위가 명확해졌다.
