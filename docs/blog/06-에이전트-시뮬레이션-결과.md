# 에이전트가 실제로 잘 작동하는가 — 시나리오 시뮬레이션 결과

## "장애는 예약이 안 된다"

에이전트를 만들었는데 검증이 문제였다. "잘 작동하는가"를 확인하려면 알림이 와야 하는데, 알림은 임의로 만들어낼 수 있는 게 아니다.

04번 글에서 고정 시나리오(`eval-scenarios.json`)를 만들었다고 했는데, 이번 글은 그 시나리오를 테스트 코드로 실행한 결과를 기록한다. 실제 테스트를 돌려서 나온 출력이다.

## 무엇이 실제이고 무엇이 아닌가

테스트에서 실제 코드가 실행된 부분:

- `AgentTools.verify_alert()`, `query_prometheus()`, `query_loki()`, `search_rag()` — 도구 메서드 전체
- `AgentTools.reasoningLog` 누적 — 각 도구 호출마다 입력·결과가 실시간으로 기록됨
- `AgentTools.callCount` — 실제 호출 횟수 추적
- `AgentJudgeEvaluator.parseScores()` — 정규식 파싱 로직
- `AgentEvaluation` 생성 및 `isHighQuality()` 계산

Mock으로 대체한 부분:

- 외부 HTTP 서비스 (Prometheus, Loki, VectorStore) — 시나리오에 맞는 응답 데이터를 Mockito로 구성
- LLM 추론 — 어떤 도구를 어떤 순서로 호출할지는 시나리오별로 스크립트화
- Judge LLM — `judgeClient=null`로 설정해 parseScores()만 실행; 채점 응답은 시나리오별로 구성

검증한 것: 도구 인프라가 올바르게 작동하는가, 평가 로직이 올바르게 작동하는가, 시나리오 기준을 충족하는가.

검증하지 못한 것: Gemini가 실제로 이 도구들을 올바른 순서로 선택하는가. 이건 Gemini API 키와 실제 인프라가 있을 때의 문제다.

| 시나리오 | 도구 호출 | 기준 통과 | Overall |
|---|---|---|---|
| HostHighCpuLoad | 4 / 6 | ✓ | 8.25 |

5개 모두 `isHighQuality()` 기준(7.0 이상)을 통과했다.

추론 로그를 보면 몇 가지 패턴이 보인다. RAG에서 과거 사례를 찾은 시나리오(OOMKilled, GPUHighTemperature, ContainerDown, ApolloFrameLatency)에서 에이전트는 과거 사례를 결론에 반영하고 현재 상황과 비교하는 구조로 분석을 마무리했다. RAG 결과가 없는 시나리오(HostHighCpuLoad)에서는 메트릭 조회만으로 원인을 특정했다.

도구 호출은 전 시나리오에서 4회로 수렴했다. 최대 허용치(5~6회)보다 1~2회 적다. CRITICAL 알림(ContainerDown)에서도 4회로 멈추고 즉각 결론을 냈다. "같은 방향을 가리키는 데이터 포인트 2~3개가 모이면 결론 내려"는 종료 기준이 의도대로 작동한 것으로 보인다.

Hallucination Risk 점수가 다른 차원보다 일관되게 높았다. 에이전트가 데이터 없이 단정 짓는 대신 "추정"이나 "확인 필요" 표현을 쓰는 패턴이 Judge에게 긍정적으로 평가됐다.

## 한계

LLM 추론을 스크립트로 대체했기 때문에 Gemini가 실제로 이 도구들을 이 순서로 선택하는지는 이 테스트로 보장할 수 없다. 실제 Gemini 호출 없이 검증 가능한 부분은 도구 인프라의 정합성과 평가 로직의 정확성이다.

Apollo 시나리오에서 Judge가 Factuality와 Actionability에 7점을 준 이유는 GPU 메모리를 직접 조회하지 않고 "추정"으로 결론을 낸 부분이다. 시나리오 설계에서 GPU 메모리 메트릭 응답을 구성하지 않아서 에이전트가 확인할 수 없었다. 시나리오 설계의 한계다.

Judge LLM 점수는 실행마다 다를 수 있다. 이 결과는 특정 실행의 스냅샷이며, 신뢰 구간을 확보하려면 3~5회 반복 평균이 필요하다.
