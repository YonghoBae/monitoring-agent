# AlertService가 너무 많은 걸 알고 있었다

## 처음 구조

알림이 오면 분석하고 Discord로 전송하는 흐름은 직관적으로 `AlertService`에 넣었다.

```java
@Transactional
public AlertEvent createAlert(String level, String message) {
    AlertEvent saved = alertEventRepository.save(alert);
    alertVectorService.indexAlert(saved);
    if (shouldNotify(level)) {
        runAgentPipeline(saved);  // Gemini 분석 + Discord 전송
    }
    return saved;
}
```

코드는 간단해 보였다. 그런데 `AlertService`의 생성자를 보면 이야기가 달라진다.

```java
public AlertService(
    AlertEventRepository,
    AlertVectorService,
    DiscordNotificationService,
    ReActAgent,
    OrchestratorAgent,
    AlertVerifier,
    AlertPlaybook,
    CommandExecutorService,
    ObjectMapper
)
```

알림을 저장하는 서비스가 AI 에이전트, Discord 전송, 명령 실행까지 직접 의존하고 있었다. "알림 저장 서비스"라는 이름과 실제 책임이 달랐다.

## 발견한 문제

더 구체적인 문제가 있었다. `createAlert()`는 `@Transactional`인데, 그 안에서 Gemini API를 호출한다.

```
@Transactional 시작
  → alertEventRepository.save()       ← DB 커넥션 획득
  → alertVectorService.indexAlert()
  → alertVerifier.verify()            ← Prometheus API (외부 호출)
  → reActAgent.run()                  ← Gemini API (수 초~수십 초)
  → discordNotificationService.send() ← Discord API (외부 호출)
@Transactional 끝                     ← 그제서야 커넥션 반납
```

트랜잭션이 열려있는 동안 DB 커넥션이 점유된다. Gemini 분석이 오래 걸릴수록, 알림이 동시에 여러 개 들어올수록, 커넥션 풀이 고갈될 가능성이 높아진다. 지금은 혼자 쓰는 서버라 문제가 없지만, 구조적으로 잘못된 상태였다.

## 분리 기준

"저장"과 "분석"은 시점이 달라야 한다.

- **저장**: 알림 데이터를 DB에 커밋하는 작업. 빠르게 완료되어야 한다.
- **분석**: 저장이 완료된 후, AI가 비동기로 처리하는 작업.

Spring의 `ApplicationEventPublisher`와 `@TransactionalEventListener`가 이 시점 분리를 정확히 표현한다.

```java
// AlertService — 저장만
@Transactional
public AlertEvent createAlert(String level, String message) {
    AlertEvent saved = alertEventRepository.save(alert);
    alertVectorService.indexAlert(saved);
    if (shouldNotify(level)) {
        eventPublisher.publishEvent(new AlertCreatedEvent(saved.getId()));
    }
    return saved;  // 여기서 트랜잭션 끝
}
```

```java
// AlertPipelineListener — 커밋 후 별도 스레드에서 실행
@Async
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onAlertCreated(AlertCreatedEvent event) {
    alertAnalysisChain.run(event.alertId());
}
```

`AFTER_COMMIT`은 트랜잭션이 성공적으로 커밋된 후에만 리스너를 실행한다. `@Async`는 별도 스레드에서 실행해 `AlertService`의 응답을 블로킹하지 않는다. 두 조합으로 Gemini 호출 중에는 DB 커넥션이 없다.

## Chain Workflow로 파이프라인 명시

분리 후 파이프라인 로직을 `AlertAnalysisChain`에 옮기면서, 기존에 하나의 메서드에 뭉쳐있던 단계를 명시적으로 표현했다.

```java
public void run(Long alertId) {
    AlertEvent alert = alertEventRepository.findById(alertId).orElseThrow();

    // Step 1: Verify
    VerificationResult verification = alertVerifier.verify(alert);
    alert.setVerificationStatus(verification.status().name());
    alertEventRepository.save(alert);

    if (verification.status() == VerificationResult.Status.STALE) {
        return;  // 알림이 이미 해소됨 — 여기서 종료
    }

    // Step 2: Playbook
    ActionRecommendation recommendation = alertPlaybook.lookup(alert.getAlertName());

    // Step 3: Analyze
    AgentResult agentResult = runAgent(alert, recommendation);
    alert.setAnalysisResult(agentResult.conclusion());
    alertEventRepository.save(alert);

    // Step 4: Authorize
    authorize(alert, agentResult, verification, recommendation);
}
```

Anthropic의 "Building Effective Agents"에서 설명하는 Chain Workflow 패턴이다. 각 단계가 독립적이고, 이전 단계의 출력이 다음 단계의 입력이 된다.

이전 코드에서 못 했던 것도 자연스럽게 들어갔다. Prometheus 확인 결과 알림이 이미 해소된 경우(`STALE`), 기존 코드는 Gemini 분석을 불필요하게 돌리고 있었다. 이제는 Step 1에서 바로 종료한다.

각 `save()` 호출은 Spring Data의 자체 트랜잭션 안에서 즉시 완료된다. Step 3의 Gemini 호출 전후로 커넥션이 잡혀있지 않다.

## 결과

```
Before
AlertService(9개 의존성):
  save → Gemini(수십초) → Discord  ← @Transactional이 전 과정을 감쌈

After
AlertService(4개 의존성):
  save → publishEvent → 커밋

AlertPipelineListener (별도 스레드, 커밋 후):
  → AlertAnalysisChain
      Step1: verify  → save
      Step2: playbook
      Step3: Gemini  → save   ← 커넥션 없음
      Step4: Discord
```

`AlertService`가 에이전트를 몰라도 된다. 알림이 저장됐다는 사실만 알리면 된다.
