# 도구를 두 종류로 나눈 이유

## "도구 하나로 다 처리하면 안 됐나?"

처음 도구 설계를 할 때 모든 도구를 `AgentTools` 하나에 넣으려 했다. `verify_alert`, `query_prometheus`, `query_loki`, `search_rag`, `web_search`에 명령 실행까지. 하나의 클래스에 모든 기능을 모아두면 관리하기 편하다.

그런데 이 프로젝트에서 에이전트는 두 가지 다른 맥락에서 실행된다.

- **ReAct 에이전트**: 알림이 오면 자동으로 실행. 분석이 목적. 운영자가 보고 있지 않아도 돌아간다.
- **ConversationAgent**: 운영자가 Discord에서 직접 대화. 운영자의 요청으로 실행된다.

두 에이전트가 같은 도구를 공유하면 ReAct 에이전트도 명령 실행 도구를 가지게 된다. 알림이 왔을 때 자동으로 컨테이너를 재시작하거나 명령을 실행할 수 있다는 뜻이다. 혼자 운영하는 서버라도, 에이전트가 분석 단계에서 실수로 혹은 과잉 판단으로 명령을 실행하는 건 허용하고 싶지 않았다.

에이전트가 스스로 결론을 내리는 것과, 그 결론에 따라 직접 실행하는 것 사이에 경계를 두기로 했다.

## 도구 분리

`AgentTools`는 읽기 전용으로 유지했다.

```java
// AgentTools — 모든 조회는 이쪽
verify_alert(alertName, labelsJson)   // Prometheus 현재 상태 확인
query_prometheus(promql, timeRange)   // 메트릭 조회
query_loki(containerName, timeRange)  // 로그 조회
search_rag(query)                     // 과거 사례 벡터 검색
```

`ConversationTools`는 ConversationAgent 전용으로 만들고, `execute_command` 하나만 넣었다.

```java
// ConversationTools — 실행은 이쪽, 운영자 확인이 필요
execute_command(command, reason)      // 명령 실행 제안 (즉시 실행이 아님)
```

중요한 건 `execute_command`가 실제로 명령을 실행하지 않는다는 점이다. 실행이 아니라 "제안"이다.

```java
@Tool(description = "명령 실행을 제안한다. 실제 실행 전 운영자의 확인(yes/no)이 필요하다.")
public String execute_command(String command, String reason) {
    if (!commandExecutorService.isAllowed(command)) {
        return "거부됨: 허가되지 않은 명령어입니다. 허용 형식: docker restart <container-name>";
    }

    pendingApprovalStore.store(command, alertId, channelId);
    return "PENDING_CONFIRM: 운영자에게 실행 여부를 확인하세요. 운영자가 'yes'로 확인하면 실행됩니다.";
}
```

에이전트가 `execute_command`를 호출하면 `PendingApprovalStore`에 명령이 대기 상태로 저장된다. Discord에서 운영자가 "yes"를 입력해야 그때 실제로 실행된다.

```
에이전트: docker restart nginx를 실행하는 것을 제안합니다.
          (이유: 로그에서 OOMKilled 확인. 재시작으로 임시 복구 가능)
          실행하려면 'yes', 취소하려면 'no'를 입력하세요.

운영자: yes

시스템: ✅ 실행 완료: docker restart nginx
```

## 명령 실행 흐름의 전체 경계

에이전트에서 실제 명령 실행까지 경계가 여러 겹이다.

**첫 번째 경계: `execute_command` 도구의 허용 여부**

`ConversationTools.execute_command`가 호출되면 먼저 `CommandExecutorService.isAllowed()`를 통과해야 한다.

```java
// CommandExecutorService
private static final Pattern ALLOWED_CONTAINER_NAME =
        Pattern.compile("^[a-zA-Z0-9][a-zA-Z0-9_.\\-]*$");

public boolean isAllowed(String command) {
    String[] parts = command.trim().split("\\s+");
    return parts.length == 3
            && "docker".equals(parts[0])
            && "restart".equals(parts[1])
            && ALLOWED_CONTAINER_NAME.matcher(parts[2]).matches();
}
```

`docker restart <container-name>` 형식만 허용한다. 에이전트가 프롬프트 인젝션을 당하거나 예상치 못한 명령을 생성하더라도 이 검증을 통과하지 못하면 거부된다.

**두 번째 경계: PendingApprovalStore와 운영자 확인**

검증을 통과한 명령도 바로 실행되지 않는다. `PendingApprovalStore`에 30분 TTL로 저장되고, Discord에서 "yes"가 들어와야 실행된다.

```java
// PendingApprovalStore
public void store(String command, Long alertId, String channelId) {
    Instant expiresAt = Instant.now().plusSeconds(ttlMinutes * 60);
    store.put("channel:" + channelId, new PendingApproval(command, alertId, expiresAt, channelId));
}
```

채널 ID를 키로 저장하기 때문에, 같은 채널에서 "yes"를 입력한 사람만 해당 채널의 대기 명령을 실행할 수 있다.

**세 번째 경계: DiscordBotListener의 라우팅**

Discord 메시지는 `DiscordBotListener`에서 우선순위에 따라 라우팅된다.

```java
// DiscordBotListener.onMessageReceived()
if (content.equalsIgnoreCase("yes") || content.equals("확인")) {
    handleYes(event, channelId);
    return;
}
if (content.equalsIgnoreCase("no") || content.equals("취소")) {
    handleNo(event, channelId);
    return;
}
// 그 외 → 대화로 처리
handleConversation(event, channelId, content);
```

"yes/no" 처리 경로와 대화 처리 경로가 분리되어 있다. 대화 흐름 중에 에이전트가 생성한 응답 텍스트가 우연히 "yes"처럼 보이더라도, 운영자가 직접 채널에 "yes"를 입력하는 것과는 다른 경로다. 에이전트의 출력이 스스로 실행을 트리거하는 건 불가능하다.

## 도구 분리가 가져온 다른 이점

두 도구 클래스를 분리하면서 `ConversationTools`의 생성자에 `channelId`와 `alertId`를 넣게 됐다.

```java
public ConversationTools(CommandExecutorService commandExecutorService,
                         PendingApprovalStore pendingApprovalStore,
                         String channelId,
                         @Nullable Long alertId) { ... }
```

`ConversationTools`가 생성되는 시점에 컨텍스트(어느 채널, 어떤 알림)가 이미 고정된다. `execute_command`가 호출될 때 이 정보를 인자로 다시 받을 필요가 없다. 도구 메서드의 시그니처가 단순해지고, 에이전트가 채널 ID를 잘못 넘길 가능성도 없어진다.

## 한계

현재 허용된 명령은 `docker restart` 하나다. 운영 범위가 넓어지면 허용 명령도 늘어야 하는데, 명령이 늘수록 `isAllowed()` 로직이 복잡해지고 검증 실수 가능성도 높아진다. 지금 구조에서 새로운 명령 유형을 추가하려면 정규식 패턴과 파싱 로직을 함께 수정해야 한다. 단순히 목록에 추가하는 방식보다 오류가 생기기 쉽다.
