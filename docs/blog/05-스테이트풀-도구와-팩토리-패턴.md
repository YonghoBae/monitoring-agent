# 스테이트풀 도구와 팩토리 패턴

## "Spring에서 @Tool 메서드를 Bean으로 만들면 안 됐나?"

Spring AI에서 에이전트 도구를 등록하는 방법은 간단하다. `@Tool` 어노테이션을 붙인 메서드가 있는 객체를 `.tools()`에 넘기면 된다.

```java
chatClient.prompt()
        .system(systemPrompt)
        .user(alertDescription)
        .tools(agentTools, webSearchTool)  // @Tool 메서드가 있는 객체
        .call()
        .content();
```

자연스럽게 `AgentTools`를 Spring Bean으로 만들려고 했다. `@Service` 어노테이션 하나면 Spring이 관리해주고, 의존성 주입도 자동으로 된다. 그게 Spring 방식이다.

그런데 `AgentTools`에는 이런 필드가 있었다.

```java
public class AgentTools {
    // 현재 실행 중인 ReAct 루프의 추론 로그
    private final StringBuilder reasoningLog = new StringBuilder();
    private int callCount = 0;
    ...
}
```

## 문제

Spring Bean은 기본적으로 싱글톤이다. 애플리케이션이 뜰 때 인스턴스가 하나 생성되고, 모든 요청이 그 인스턴스를 공유한다.

`AgentTools`가 싱글톤이라면, `reasoningLog`도 싱글톤이다. 알림 A의 ReAct 루프가 `verify_alert`를 호출하면 `reasoningLog`에 기록이 남는다. 이 상태에서 알림 B의 루프가 시작되면, 알림 A의 기록이 이미 들어있는 `reasoningLog`를 그대로 이어받는다.

결론과 추론 로그가 뒤섞인다. ReActAgent가 루프 실행 후 `agentTools.getReasoningLog()`를 읽어서 분석 근거를 기록하는데, 여기에 다른 알림의 도구 호출 내역이 섞이는 것이다.

`callCount`도 마찬가지다. 루프마다 "이번 분석에서 도구를 몇 번 호출했는가"를 세는데, 싱글톤이면 이전 루프의 카운트가 누적된다.

두 알림이 동시에 처리되면 더 심각하다. 두 루프가 같은 `reasoningLog`에 동시에 쓰면 내용이 예측 불가능하게 뒤섞인다. `StringBuilder`는 스레드 안전하지 않다.

## 요청별 인스턴스가 필요하다

해결 방향은 단순하다. ReAct 루프를 실행할 때마다 새 `AgentTools` 인스턴스를 만들면 된다. 인스턴스가 새로 생성될 때 `reasoningLog`는 빈 StringBuilder로, `callCount`는 0으로 초기화된다.

Spring에서 요청마다 새 인스턴스를 만드는 방법으로는 `@Scope("prototype")`이 있다. 그런데 이 방식은 주입 시점에 인스턴스가 생성되기 때문에, 인스턴스 생성 시점을 명시적으로 제어하기 어렵다. ReActAgent가 루프를 실행할 때 정확히 새 인스턴스를 받아야 하는데, 프로토타입 빈은 그 제어가 불명확해진다.

팩토리 패턴을 선택했다.

```java
// AgentToolsFactory — @Component (싱글톤)
@Component
public class AgentToolsFactory {

    private final AlertVerifier alertVerifier;
    private final PrometheusQueryService prometheusQuery;
    private final LokiQueryService lokiQuery;
    private final AlertVectorService vectorService;

    // 생성자 주입 생략

    public AgentTools createAgentTools() {
        return new AgentTools(alertVerifier, prometheusQuery, lokiQuery, vectorService);
    }
}
```

`AgentToolsFactory`는 싱글톤 Bean이다. 팩토리 자체의 의존성(`AlertVerifier` 등)은 Spring이 주입해준다. `createAgentTools()`를 호출하면 그 시점에 새 `AgentTools` 인스턴스가 생성된다.

```java
// ReActAgent.runInternal()
AgentTools agentTools = agentToolsFactory.createAgentTools(); // 매 루프마다 새 인스턴스
String conclusion = chatClient.prompt()
        .tools(agentTools, webSearchTool)
        .call()
        .content();

String reasoningChain = agentTools.getReasoningLog(); // 이 루프의 기록만 있음
int iterationCount = agentTools.getCallCount();       // 이 루프의 카운트만 있음
```

Reflection으로 재분석할 때도 새 인스턴스를 만든다.

```java
AgentTools retryTools = agentToolsFactory.createAgentTools(); // 재분석도 새 인스턴스
```

## ConversationTools의 추가 컨텍스트

`ConversationTools`도 같은 이유로 팩토리 패턴을 쓴다. 그런데 여기엔 한 가지가 더 있다.

```java
// ConversationTools
public class ConversationTools {
    private final String channelId;
    @Nullable
    private final Long alertId;
    ...
}
```

`channelId`와 `alertId`가 인스턴스 필드다. `execute_command`가 호출될 때 어느 채널의 어떤 알림에 대한 명령인지를 이미 인스턴스가 알고 있어야 한다. 이 정보를 `execute_command`의 인자로 받으면 에이전트가 채널 ID를 직접 알아야 하는데, 에이전트가 그 정보를 가질 이유가 없다.

```java
// ConversationToolsFactory
@Component
public class ConversationToolsFactory {

    private final CommandExecutorService commandExecutorService;
    private final PendingApprovalStore pendingApprovalStore;

    public ConversationTools create(String channelId, @Nullable Long alertId) {
        return new ConversationTools(commandExecutorService, pendingApprovalStore, channelId, alertId);
    }
}
```

팩토리 호출 시점에 `channelId`와 `alertId`를 넘기면, 그 이후 `execute_command`가 호출될 때 이 값들이 이미 인스턴스 안에 있다. 에이전트가 "이 명령은 어느 채널의 것입니까"를 알 필요가 없어진다.

```java
// ConversationAgent.chat()
ConversationTools convTools = conversationToolsFactory
        .create(session.channelId(), session.alertId()); // 컨텍스트 고정
```

생성 시점에 컨텍스트가 캡처된다는 점에서 클로저와 비슷한 발상이다.

## 패턴의 핵심

팩토리 패턴이 해결한 건 두 가지다.

첫째, 상태 격리. 각 ReAct 루프는 자신만의 `reasoningLog`와 `callCount`를 가진다. 다른 루프와 섞일 여지가 없다.

둘째, 생성 시점 제어. `new AgentTools()`가 호출되는 시점이 코드에서 명시적으로 보인다. 인스턴스가 언제 만들어지는지 추적하기 쉽다.

Spring의 DI 컨테이너에 모든 것을 맡기는 것이 항상 최선은 아니다. 요청별 생명주기가 필요한 객체는 팩토리로 직접 관리하는 게 더 명확할 때가 있다.

## 한계

현재 `AgentTools`는 Spring Bean이 아니기 때문에 Spring의 AOP가 적용되지 않는다. 예를 들어 `@Transactional`이나 트레이싱 어노테이션이 동작하지 않는다. 도구 호출에 세밀한 모니터링이 필요해질 경우, 팩토리에서 프록시를 직접 생성하거나 다른 방법을 써야 한다.
