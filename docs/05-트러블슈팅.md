# 트러블슈팅

개발 과정에서 실제로 겪었던 문제들과 해결 과정을 기록한다. git 히스토리로 추적 가능한 문제를 중심으로 정리했다.

## 1. GraalVM 플러그인 제거 후 빌드 실패

**커밋 근거:** `ab1c487 fix: Java 17 → 21 버전 수정`

### 상황

GraalVM 네이티브 이미지 빌드 플러그인이 불필요하다고 판단해 `build.gradle`에서 제거했다.

```gradle
// 제거한 라인
id 'org.graalvm.buildtools.native' version '0.10.6'
```

### 문제

플러그인을 제거하자 빌드가 실패했다.

```
FAILURE: Build failed with an exception.
* What went wrong: Task 'processTestAot' not found in root project.
```

### 원인

GraalVM 플러그인이 제공하는 `processTestAot` 태스크를 `build.gradle` 하단에서 참조하고 있었다. 플러그인만 제거하고 해당 태스크 설정 블록은 그대로 남아 있었다.

```gradle
// 플러그인 제거 후에도 남아있던 코드
tasks.named("processTestAot") {
    enabled = false
}
```

### 해결

플러그인과 함께 `processTestAot` 태스크 블록도 제거했다.

### 교훈

Gradle 플러그인은 태스크, 설정 블록, 의존성 세 가지를 함께 관리한다. 플러그인을 제거할 때 해당 플러그인이 제공하는 태스크를 참조하는 코드가 없는지 전체를 확인해야 한다.

## 2. 서비스 생성자 변경 후 테스트 컴파일 실패

**커밋 근거:** `6215431 test: 에이전트 파이프라인 및 Discord 서비스 테스트 수정`

### 상황

Discord 양방향 대화 기능 구현을 위해 `DiscordNotificationService`에 새 의존성을 추가했다.

```java
// 기존 생성자
public DiscordNotificationService(JDA jda, AlertEventRepository repo) { ... }

// 변경된 생성자
public DiscordNotificationService(JDA jda, AlertEventRepository repo,
    String botChannelId, ConversationSessionStore sessionStore) { ... }
```

### 문제

```
error: constructor DiscordNotificationService in class DiscordNotificationService
cannot be applied to given types
  required: JDA, AlertEventRepository, String, ConversationSessionStore
  found: JDA, AlertEventRepository
```

### 원인

`DiscordNotificationServiceTest`가 이전 2-인자 생성자로 객체를 생성하고 있었다. 프로덕션 코드 생성자가 바뀌었지만 테스트 코드는 그대로였다.

### 해결

테스트 파일에 mock 추가 및 생성자 인자를 맞췄다.

```java
// 추가된 mock
@Mock private ConversationSessionStore sessionStore;

// 생성자 인자 추가
new DiscordNotificationService(jda, repo, "test-channel-id", sessionStore)
```

### 교훈

생성자 기반 의존성 주입은 안전하지만(누락된 의존성을 컴파일 시점에 감지), 테스트 코드와의 동기화를 항상 확인해야 한다. `@InjectMocks`를 사용하면 Mockito가 생성자를 자동으로 매칭해주지만 직접 생성자를 호출하는 테스트는 수동으로 업데이트해야 한다.

## 3. Discord 봇 메시지 무반응

### 상황

JDA 봇을 Discord 서버에 초대하고 채널에 메시지를 보냈는데 `onMessageReceived` 이벤트가 발생하지 않았다. Java 애플리케이션 로그에도 아무 출력이 없었다.

### 문제

봇이 메시지를 받고 있는지 확인할 방법이 없었다. JDA 자체 로그에도 Intent 관련 경고가 없었다.

### 원인

Discord는 2022년부터 **Privileged Gateway Intent** 정책을 시행한다. 봇이 메시지 내용을 읽으려면 Discord Developer Portal에서 `MESSAGE CONTENT INTENT`를 명시적으로 활성화해야 한다.

JDA 코드에서는 `GatewayIntent.MESSAGE_CONTENT`를 지정했지만, Developer Portal에서 활성화하지 않으면 이벤트 자체가 전달되지 않는다.

```java
// JDA 코드 — 이것만으로는 부족
JDABuilder.createDefault(token)
    .enableIntents(GatewayIntent.GUILD_MESSAGES, GatewayIntent.MESSAGE_CONTENT)
    .build();
```

### 해결

Discord Developer Portal → 봇 설정 → **Privileged Gateway Intents** → `MESSAGE CONTENT INTENT` 활성화

### 교훈

외부 서비스(Discord, GitHub 등)와 연동할 때 SDK 코드와 콘솔/포털 설정이 양쪽 다 맞아야 한다. SDK에서 설정했다고 자동으로 활성화되지 않는다. JDA가 Intent 미활성화 상태를 로그로 경고하지 않아서 디버깅이 어려웠다.

## 4. Docker 이미지 최적화 — 이미지 크기 55% 감소, 재빌드 98% 단축

### 상황

초기 Dockerfile은 Ubuntu 기반 JRE와 `docker.io` 전체 패키지를 설치하고 있었다.

```dockerfile
# 초기 Dockerfile
FROM eclipse-temurin:21-jre
RUN apt-get update && apt-get install -y docker.io && rm -rf /var/lib/apt/lists/*
RUN ./gradlew bootJar -x test
```

### 문제

- 이미지 크기 **728MB** — 불필요하게 큰 이미지
- 빌드 시간 **83초** — 소스 변경 없이도 매번 Gradle 의존성 재다운로드

### 원인 및 해결 과정

**문제 1: 이미지 크기**

`eclipse-temurin:21-jre`는 Ubuntu 기반으로 약 220MB다. Alpine Linux 기반 이미지(`eclipse-temurin:21-jre-alpine`)는 약 80MB다. 런타임 이미지는 컴파일러가 필요 없으므로 Alpine이 충분하다.

`docker.io`는 Docker 데몬을 포함한 전체 패키지다. 이 서비스에서는 컨테이너를 **재시작**하는 명령만 필요하다. Docker CLI(`docker-cli`)만 설치하면 충분하다.

```dockerfile
# 변경 후
FROM eclipse-temurin:21-jre-alpine
RUN apk add --no-cache docker-cli
```

**문제 2: 빌드 시간 (Gradle 의존성 재다운로드)**

Docker 빌드는 레이어 캐시를 활용하지만, `RUN ./gradlew bootJar`가 실행될 때마다 Gradle이 `~/.gradle` 디렉토리에 의존성을 다운로드한다. 이 디렉토리는 빌드 컨테이너에 존재하므로 빌드가 끝나면 사라진다.

Docker BuildKit의 `--mount=type=cache`를 사용하면 특정 디렉토리를 빌드 간에 유지할 수 있다.

```dockerfile
RUN --mount=type=cache,target=/root/.gradle \
    ./gradlew bootJar -x test --build-cache
```

### 실측 결과

| 항목 | 이전 | 이후 | 개선 |
|------|------|------|------|
| 이미지 크기 | 728 MB | 323 MB | **55.6% 감소** |
| 첫 빌드 시간 | 83초 | 54초 | 35% 단축 |
| 재빌드 시간 | 83초 | **1.2초** | **98.5% 단축** |

재빌드가 1.2초인 이유는 `--mount=type=cache`로 Gradle 의존성이 캐시에 그대로 남아있어 다운로드를 건너뛰기 때문이다.

> 측정 환경: 동일 머신, `--no-cache` 플래그로 Docker 레이어 캐시 제거 후 측정. 재빌드는 첫 빌드 직후 동일 코드로 측정.
